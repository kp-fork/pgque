# benchmark/xmin-horizon

Reproducible benchmark for PG-backed queues under a blocked xmin horizon.

The thesis: most published Postgres-queue benchmarks measure throughput in
steady state — empty buffer cache, no application load, no replication lag,
autovacuum unimpeded. But in production, the most common operational
failure mode of `SELECT ... FOR UPDATE SKIP LOCKED` queues is silent
degradation when the xmin horizon is held by a long-running transaction or an
unconsumed logical replication slot. VACUUM cannot reclaim the dead tuples
generated by `INSERT → UPDATE → DELETE` cycles, the queue table bloats,
and read latency climbs.

This benchmark contrasts two workloads under that condition:

- **W-skiplocked** — a generic single-table queue using `FOR UPDATE SKIP LOCKED`.
- **W-pgque** — pgque, which rotates events across `event_N` tables and
  reclaims via TRUNCATE rather than DELETE — the hot path generates zero
  dead tuples on the queue tables.

## Run

Requires Docker and `psql` + `pgbench` on the host (homebrew `libpq` is enough).

```sh
cd benchmark/xmin-horizon
make up                 # start PG17 in Docker
make run-s1             # baseline, both workloads, default 300s each
make run-s2             # RR transaction holds xmin, both workloads
make report             # aggregate into results/results.md
make down               # tear down
```

Override `DURATION` to change cell length:

```sh
DURATION=120 make run-s1 run-s2 report
```

## Scenarios

| ID  | Description |
|-----|-------------|
| S1  | Baseline. Queue under sustained load, no xmin holder. |
| S2  | A `BEGIN ISOLATION LEVEL REPEATABLE READ` transaction held open for the duration of the cell. |
| S3  | (Future) Idle logical replication slot. |

## Outputs

`results/<scenario>-<workload>/`:
- `metrics.csv` — 5s sampling of counters, bloat, xmin
- `producer.log` / `consumer.log` / `bystander.log` — pgbench output
- `final-bloat.csv` — `pg_stat_user_tables` snapshot at end of run
- `final-counters.csv` — enqueued/dequeued totals

`results/results.md` — aggregated summary table across all cells.

## Framing

The benchmark is a comparison of queue *implementations* under realistic
operational conditions, not a takedown of any specific library. `SKIP LOCKED`
is a sound primitive; the failure mode is structural to its `INSERT/UPDATE/
DELETE` lifecycle, not specific to any one implementation.
