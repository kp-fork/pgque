# R10 — adding awa to the 7-system bench

## Scope

R10 adds [awa](https://github.com/hardbyte/awa) as the 8th system on the same
50-minute schedule used in R9 (10m clean + 30m idle-in-tx + 10m recovery,
producer `-R 2000`, fresh `i4i.2xlarge` spot VMs in us-east-2 / PG 18 / NVMe).
The R7 / R8 / R9 systems (pgque, pgq, pgmq, pgmq-partitioned, river, que,
pgboss) are unchanged.

Round results: [GitLab issue note](https://gitlab.com/postgres-ai/postgresql-consulting/tests-and-benchmarks/-/issues/77#note_3304910299).

## Driver model for awa

Native Python worker, matching the convention "use each system's native API"
that R7-R9 already follow for the SQL-driven systems:

- `producers/producer_awa.py` — `awa.AsyncClient.insert(BenchJob(...))` in
  async batches of 50 every 25 ms to hit `-R 2000`.
- `consumers/consumer_awa.py` — registers a no-op handler, then
  `client.start([("bench", 4)])` for 4 worker tasks. Emits
  `NOTICE: ev ts=<sec> n=<count>` once per second so the existing
  `parse_events_consumed.py` works unchanged.
- `install/install_awa.sh` — `pip install awa-pg awa-cli` into
  `/opt/awa-venv`, then `awa migrate`.

## Runner

`runners/run_r10.sh <sys>` is a small evolution of `run_r7.sh`:

- 50m schedule (10m + 30m + 10m) instead of 90m.
- Branches on the `awa` system to launch the Python producer/consumer instead
  of pgbench.
- Calls the psql-based `tooling/idle_in_tx.sh` instead of `idle_in_tx.py`
  (no `psycopg2` dependency on the bare AMI).
- Same observability set as R9: pg_ash dump (`ash.csv`), pg_stat_statements
  snapshot (`pgss.csv`), `events_consumed_per_sec.csv`, `sys_metrics.csv`.

## Charts

- `charts/r10_throughput_chart.py` — 8-panel consumer-throughput timeline
  with phase markers and per-phase averages in panel headers.
- `charts/r10_sysmetrics_chart.py` — CPU user+sys %, NVMe write MiB/s, NVMe
  write IOPS as three time-series with all 8 systems overlaid.
- `charts/r10_ash_chart.py` — Performance-Insights-style 8-panel ASH chart,
  stacked by `wait_event_type`, colors per the
  [pg_ash convention](https://github.com/NikolayS/pg_ash/blob/main/docs/COLOR_SCHEME.md).
- `charts/r10_summary_table.py` — per-system markdown table:
  producer/consumer totals, TX-avg ev/s, true backlog
  (`producer_total - consumer_total`), TX-peak CPU%, TX-peak NVMe MiB/s.
- `charts/r10_ash_table.py` — per-system per-phase wait-event mix.

All chart scripts default `ROOT = Path("/tmp/bench_r10")` and write to
`/tmp/r10_*.png` / stdout. Adjust as needed for local re-runs.

## Headline result (R10 specific)

The R7-R9 systems reproduce. The new datapoint is awa:

- TX-avg throughput **68 ev/s** (3.4 % of offered) — the largest collapse of
  the eight systems on this workload.
- True backlog **5.33 M events** at end of run.
- ASH dominated by `Lock:transactionid` waits in Clean (34 %), TX (57 %),
  and Recovery (77 %).
- `Lock:transactionid` = a backend waiting for another backend's transaction
  to end. In a `SELECT FOR UPDATE SKIP LOCKED` design this should be near
  zero. On this workload, awa's claim path takes row locks that serialize
  across workers.

This is one workload (sustained `-R 2000` with 30 min held xmin) on one
Postgres config. Different workloads may show very different shapes; the
finding is data, not a verdict — happy to re-run with awa-author tuning
suggestions.

awa version: 0.5.6 from PyPI.
